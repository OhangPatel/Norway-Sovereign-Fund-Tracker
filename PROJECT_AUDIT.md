## 3. Database Analysis

### Current state: 3 tables in `backend/nbim.db`

**`financial_metrics`**
```sql
ticker TEXT, fetched_at DATETIME,
pe_ratio REAL, forward_pe REAL, price_to_book REAL,
dividend_yield REAL, market_cap REAL, analyst_recommendation TEXT,
target_mean_price REAL, high_52w REAL, low_52w REAL,
beta REAL, sector TEXT, industry TEXT, price REAL, change REAL
PRIMARY KEY (ticker, fetched_at)
```
→ **Problem**: PRIMARY KEY includes `fetched_at`. Every yfinance run creates 
*new rows* instead of overwriting. Right now: **5,622 rows for only 1,375 unique tickers** 
(~4 copies per company). When you merge, each ticker joins to ALL its historical rows,
producing 4× duplicates in `data.json`. That's why prices never "change" — the newest 
data is buried under old data.

**`enriched_holdings`**
```sql
-- Created by pandas .to_sql("replace") — schema types are wrong:
"Market Value(NOK)" INTEGER  -- should be REAL (large number)
"Incorporation Country" REAL  -- should be TEXT (it's a country name!)
pe_ratio TEXT               -- should be REAL (it's a number!)
```
→ **Problem**: pandas infers types badly when writing to SQLite. The `pe_ratio` column
ended up as TEXT because some values were null strings. This table is just a cache of 
`data.json` and not read by the frontend — it's the API endpoint `/api/holdings` that 
reads it, but the frontend uses `data.json` directly.

**`generated_reports`**
```sql
ticker TEXT PRIMARY KEY, description TEXT, outlook TEXT,
key_risks TEXT, generated_at DATETIME
```
→ Fine, no issues.

### The core fix needed

In `merge_and_enrich.py`, the SQL query fetches ALL rows:
```python
df_metrics = pd.read_sql("SELECT * FROM financial_metrics", conn)
```
It should only fetch the LATEST row per ticker:
```python
df_metrics = pd.read_sql("""
    SELECT * FROM financial_metrics
    WHERE fetched_at = (
        SELECT MAX(fetched_at) FROM financial_metrics m2
        WHERE m2.ticker = financial_metrics.ticker
    )
""", conn)
```
Or better — change the PRIMARY KEY to just `ticker` so `INSERT OR REPLACE` always 
overwrites the previous value instead of appending a new row (see §6 action items).

### SQLite vs. MySQL vs. MongoDB

**Keep SQLite.** Here's why:

| | SQLite | PostgreSQL/MySQL | MongoDB |
|---|---|---|---|
| Setup | Zero — already running | Requires server process | Requires server process |
| Use case fit | ✅ Single-user local tool | Better for multi-user web apps | Better for unstructured/document data |
| Financial data (structured rows) | ✅ Perfect fit | ✅ Also good | ❌ Overkill, adds complexity |
| Backup | Copy one file | pg_dump / mysqldump | mongodump |
| Viewing data | DB Browser for SQLite (free) | pgAdmin / DBeaver | Compass |

**Verdict**: Your data is structured (rows + columns), single-user, and the DB file 
is just a local cache. SQLite is the right tool. The problems aren't with SQLite —
they're with the schema design (PRIMARY KEY including timestamp = no deduplication).

**To view your database right now**: Download [DB Browser for SQLite](https://sqlitebrowser.org/) 
(free, Mac app). Open `backend/nbim.db` and you can browse all tables like a spreadsheet.

### The CSV situation

You're generating 3 CSV files (`fully_cleaned_dataset_with_reasons.csv`, 
`holdings_with_tickers.csv`, `mega_portfolio_dataset.csv`) plus the SQLite DB.

## 4. The .env Problem

Your `.env` has `DB_PATH`, `COUNTRIES`, `MISTRAL_API`, etc. — but **none of the Python 
files read it**. The database path is hardcoded in `main.py`:
```python
DB_PATH = Path(__file__).resolve().parent.parent / "nbim.db"  # hardcoded
```

Either:
- Add `python-dotenv` to `requirements.txt` and load it at the top of `main.py`
- Or accept that it's a single-machine local tool and keep paths hardcoded (simpler)

The `MISTRAL_API` key in `.env` (which is a real API key) is also not being used —
`llm_client.py` talks to local Ollama on port 11434, not Mistral's cloud API.

## 5. Frontend Analysis

### What's working well
- Clean design system with CSS variables (dark/light mode)
- Virtualized table handles 1,400+ rows without lag
- All charts are pure SVG — no Chart.js or D3 dependency
- Component architecture is clear: each `.jsx` file = one UI concern
- Keyboard navigation (/, Esc, arrow keys)

### Issues

**A) Hardcoded "last fetched" date**
In `app.jsx`:
```javascript
return 'May 7 2026'; // hardcoded forever
```
Fix: store `fetched_at` from the data in `data.json` (take `max` of all `fetched_at` values).

**B) Fake price sparklines in the detail panel**
`detail.jsx` generates a fake 120-day price history using a seeded random number 
generator. It looks real but is made-up. Users might make decisions based on it.
Options: remove it, replace with a "price history unavailable" placeholder, or fetch 
real historical data with `yfinance.Ticker(t).history(period="6mo")`.

**C) The Babel standalone setup**
Loading React + Babel in the browser works fine for development but:
- First load is slower (browser must compile all JSX)
- No tree-shaking or bundling
- Can't use modern JS syntax (async/await causes errors in some versions)

This is fine for a personal tool. If you want faster loads, migrate to a proper 
Vite build step (but that's a bigger refactor).

**D) `data.json` is read once on load**
After running the pipeline, the new data loads automatically (we fixed this with 
`dataKey` in `app.jsx`). This is fine.

## 7. The Simplified Architecture (After Cleanup)

```
Playwright scrape           yfinance batch          Merge + export
fetch_and_clean_holding.py → fetch_yahoo_metrics.py → merge_and_enrich.py
        ↓                           ↓                        ↓
holdings_with_tickers.csv    financial_metrics          data.json  ← frontend reads this
                             (SQLite, 1 row/ticker)
```

**One rule**: `financial_metrics` keeps exactly **one row per ticker** (latest).
No history, no duplicates. If you want history later, add a `financial_metrics_history` 
table and insert into that separately.
